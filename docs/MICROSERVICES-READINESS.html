Verdict: Feasible but bimodal — the edges are service-ready; the core, the agent, and the data are coupled.

The audit confirms a sharp split: external provider boundaries and the stateless AI/analytics layer are genuinely extractable today, but the conversational agent, partner-API, and notifications reach straight into getDb(), the repos, and the transfer state machine, and the per-service-database split is the real hard part (shared Postgres, partner_id on 13/16 tables, a tickets→transfers FK). The money core must stay one transactional unit — that's correct, not a flaw.

Provider seams 9/10 · AI & analytics 9/10 Agent 3.5 · Partner-API 6 · Notifications 5.5 · Data 6/10 Ledger+Outbox+Idempotency = 1 transactional unit

↻ Ground-truth corrections to the first artifact

  • Conversational Agent: 90% → 35%. agent.ts/tools.ts call getDb() at runtime and directly instantiate createTransferRepo / createTicketRepo / createOutboxRepo to mutate the ledger + outbox, plus inline sends. It is not a stateless front door today.
  • Notifications: 90% → 55%. whatsapp.ts send functions are pure, but when to send is fused to the state machine, and several sends (flagged-transfer message, OTP, cron links) fire inline, bypassing the outbox.
  • Sanctions/KYC re-scoped UP at the boundary. The provider interfaces are excellent (9/10, real factories) — they can be external services today; only the screening gate stays inline.
  • The data split is harder than implied. Per-service DBs require breaking FKs; partner_id tenancy makes Partners a shared reference service, and tickets.transfer_id is a genuine cross-context FK.

Fault-isolation reality — "if one part fails, does the system break?"

You already have most of this. Here's what actually happens today (from the code), and how much a split would add.

Net: the monolith already exhibits ~60% of the fault isolation a clean split would have — at zero coordination cost. A split adds ~15–20% more (independent process crashes, separate Redis budgets) at ~3× operational complexity (dual heartbeats, distributed tracing, cross-service session revocation, key distribution).

Honest extraction sequence (strangler-fig)

1
Wave 1 — the real free wins (do now)
Publish @smartremit/types (zero code change) → extract the AI copilots + analytics as a package. Genuinely 9/10 ready, no DB coupling. Risk: ~nil.
2
Wave 2 — productize the provider seams
The Payment/Funding/KYC/Sanctions interfaces already let real external services slot in with no call-site change. This is where "independent vendor + many-team" value is highest. Trigger: a vendor swap or a compliance team.
3
Wave 3 — refactor before you cut
Before extracting the Agent / Partner-API / Notifications: move their DB writes behind a tool/RPC boundary, route the inline sends through the outbox, and split getStore(). This is real work — don't skip it. Trigger: agent-iteration speed or a scaling wall.
★
Always — keep Partners shared & the Money Core one DB
Per-service databases break FKs into eventual consistency; the tickets→transfers link and partner_id tenancy mean Partners stays a shared reference, and the ledger never splits.

Bottom line (ground truth): realistic readiness is ~58%, not 78% — and it's bimodal. The edges are genuinely service-ready (provider interfaces, AI, analytics — extract those and you get most of the "independent teams / scale" value cheaply). The middle is coupled (agent, partner-API, notifications all reach into getDb()/repos/the state machine and need refactoring first). The core is deliberately one transactional unit, and the data layer is the true hard part (shared Postgres + partner_id everywhere + the tickets→transfers FK). You already have ~60% of the fault isolation you're after; spend effort on the high-value edges and observability, not a wholesale rewrite.  ·  Built from a 5-agent code audit.

# CI/CD Pipeline — Design

**Date:** 2026-05-23
**Status:** Awaiting approval
**Owner:** SendHome

## Why

Today, every deploy is a manual `vercel --prod` from the laptop. The branch
`feat/sendhome-prototype` has accumulated 50+ commits with no CI gate: nothing
catches a TypeScript regression, a failing test, or a broken build before it
hits production. We've shipped at least two runtime bugs (the
"Functions cannot be passed to Client Components" crash, a typosquat URL
generated by Kimi) that an automated check would have caught.

The goal of this batch is to make every future change shippable safely without
slowing down the iteration loop that's been working well.

## Scope

In:

1. A real GitHub repository as the source of truth (no remote today).
2. A `main` branch as the deploy target (today: `master` empty,
   `feat/sendhome-prototype` everything).
3. A GitHub Actions workflow that runs typecheck + lint + tests + build on
   every pull request and every push to `main`.
4. Vercel's native GitHub integration so every push to `main` ships to
   production and every PR gets a unique preview URL.
5. Branch protection on `main` requiring a passing CI run and a PR.
6. A Playwright smoke test for the dashboard's login → overview → analytics
   flow (the kind of bug that escaped us last time).
7. README + CLAUDE.md updates documenting the new workflow.

Out:

- Staging environment. Vercel preview deploys per PR already give us a
  per-branch URL with isolated state (we can choose to point each preview at
  the same Upstash DB or use a per-branch one later).
- Monorepo / package split.
- Release notes automation. Manual tagging is fine for now.
- Visual regression / Lighthouse / accessibility CI. Lane-A-but-later.
- Snyk / Dependabot. Easy to add once the basic pipeline is green.

## Architecture

```
Developer
   │
   │ git push feature-branch
   ▼
GitHub (origin)
   ├── PR opened ─────────► GitHub Actions (PR workflow)
   │                            └── typecheck · lint · test · build
   │
   ├── PR opened ─────────► Vercel GitHub App
   │                            └── Preview deploy → unique URL
   │
   │ Reviewer approves + checks green
   │ Merge PR (squash)
   ▼
main branch
   ├──────────────────────► GitHub Actions (main workflow)
   │                            └── typecheck · lint · test · build
   │
   └──────────────────────► Vercel GitHub App
                                └── Production deploy → claude-payments.vercel.app
```

Two independent things run on each event: GitHub Actions for *checks*, the
Vercel GitHub App for *deploys*. Neither blocks the other in terms of
execution, but branch protection requires the GitHub Actions check to pass
before the merge button enables.

## Components

### Repo & branches

- Create `Nagavenkatasai7/claude-payments` (public; can be private later).
- Push `feat/sendhome-prototype` to `origin/main`. The old `master` is
  abandoned (it only has the initial scaffolding commit) — we won't push it.
- Delete the local `master` after `main` is pushed.

### Scripts (in `package.json`)

We need a few small additions so the workflow doesn't have to know our
tooling:

| Script | Command | Purpose |
|---|---|---|
| `typecheck` | `tsc --noEmit` | Fast type-only check; catches `Functions cannot be passed` regressions earlier than a full build |
| `lint` | `eslint . --max-warnings 0` | Runs ESLint with the Next.js shared config |
| `test` | `vitest run --passWithNoTests` | Already present |
| `build` | `next build` | Already present |

Next 16 removed the bundled `next lint` command in favour of running ESLint
directly. We add a minimal `eslint.config.mjs` extending
`eslint-config-next`. No new rules — we accept the Next defaults.

### GitHub Actions workflow

One workflow, `.github/workflows/ci.yml`, runs on `pull_request` and
`push: [main]`. Single job, four steps after setup:

1. Checkout
2. Setup Node 24 (matches local + Vercel default)
3. `npm ci`
4. `npm run typecheck`
5. `npm run lint`
6. `npm test`
7. `npm run build`

Build is included to catch Next.js-specific errors that escape `tsc` (server
component boundary, route export shape, etc.). Build is the slowest step
(~30s); ordered last so faster failures fail fast.

No secrets are needed — none of these steps call Redis, Ollama, or Meta. The
build runs without env vars because we don't read any at build time.

### Vercel ↔ GitHub integration

Connect the existing Vercel project to the new GitHub repo through Vercel's
dashboard. Once connected:

- Push to `main` ⇒ production deploy.
- Push to any other branch ⇒ preview deploy with its own URL.
- Removing the manual `vercel --prod` flow stops the
  "deploy" classifier gate (it was a stopgap for unprompted production
  pushes; with GitHub-driven deploys, the gate is "did this PR get merged",
  which is enforced by branch protection).

Vercel pulls env vars from its own dashboard, so no Actions secrets are
needed for the deploy itself.

### Branch protection

On `main`, via `gh api`:

- Require pull request before merging (1 approval is overkill for a solo
  project; set to 0 approvals, but still require the PR — this prevents
  accidental direct pushes).
- Require status check `CI / ci` to pass.
- Require branches to be up to date before merging.
- No force push, no deletion.

### Playwright smoke test

Tiny, single-spec install. Plays the role of "would this site actually load
for a user right now":

- Test target: a preview URL passed via `BASE_URL` env var.
- Steps: open `/login`, log in with seeded admin, see "Overview", click
  "Analytics", see a chart canvas, click "Transactions", see the table.
- Runs in a *second* GitHub Actions job that depends on the Vercel preview
  being ready. Vercel exposes the URL via a deployment-status webhook;
  rather than implementing webhook plumbing in this batch, we run Playwright
  only on `push: [main]` against the production URL after the Vercel deploy
  succeeds. The PR workflow stays type/lint/test/build only.
- Total install + run < 60s, single browser (Chromium).

If this proves flaky against the live URL we'll move it to preview, but
running against prod after deploy still catches "site is down" within a
minute of every release.

## Data flow

Nothing new at runtime. The change is entirely in how code reaches Vercel:

- Before: `vercel --prod` from laptop, no checks.
- After: `git push` → GitHub Actions runs checks → if green, merge PR →
  Vercel auto-deploys → Playwright smoke checks the live site.

## Error handling

- **CI fails on a PR:** merge button stays disabled until fixed. Developer
  iterates on the PR; preview URL refreshes per push.
- **Vercel deploy fails on `main`:** Vercel keeps the previous deployment
  live; we get a notification in the GitHub PR/commit status. Roll-forward
  by reverting the offending commit (`git revert`) → push → auto-deploys.
- **Playwright smoke fails after a deploy to main:** notification only.
  We don't attempt automatic rollback in this batch — that's a Rolling
  Release feature we can add later.

## Testing strategy

- The workflow itself is tested by opening a throwaway PR with a deliberate
  type error and watching CI fail, then fixing it and watching CI pass.
- Vercel deploy is tested by squashing the rebased `feat/sendhome-prototype`
  branch into `main` and watching the production URL refresh.
- Playwright is unit-tested against the spec literally (one spec; if it
  passes locally and against prod once, we keep it).

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Migrating `feat/sendhome-prototype` → `main` rewrites history people care about | Solo project; no other contributors; we'll keep the old branch around named `archive/feat-sendhome-prototype` |
| Vercel auto-deploys break the "deploy" gate the user relied on | The new gate is "merge the PR" — same semantic intent, now visible in GitHub |
| Adding ESLint reveals dozens of pre-existing warnings | `--max-warnings 0` will fail; we accept fixing them as part of Task 2 — most likely all in `dashboard/*` JSX |
| Playwright flakiness against live prod | Single browser, short script; if flaky after 2 weeks, demote to preview-only |
| GitHub Actions minutes cost | Public repo = free; private repo on free plan = 2000 min/month, ample for our scale |

## Open questions

None. (Earlier draft asked about staging vs. preview-per-PR — settled on
preview-per-PR as Lane A. Earlier draft asked about secrets — settled on

## Acceptance criteria

- [ ] `git push origin <feature>` triggers a Vercel preview deploy with a
      reachable URL.
- [ ] Opening a PR triggers the CI workflow; merging requires it green.
- [ ] Pushing to `main` (or merging a PR into it) auto-deploys to
      `claude-payments.vercel.app` within ~2 min.
- [ ] Attempting `git push origin main` directly (without a PR) is
      rejected by branch protection.
- [ ] The Playwright smoke test runs against production after each
      deploy and reports red/green on the commit status.
- [ ] README.md describes the new contributor flow in ≤10 lines.

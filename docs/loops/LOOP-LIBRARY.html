Migration-apply guard

Prod write · approval-gated

After a migration PR merges, reads the pending drizzle migration SQL; if it is additive-only (CREATE TABLE / ADD COLUMN / CREATE INDEX) it applies it once to prod Neon with `set -a; source .env.local; set +a; npx drizzle-kit migrate`, runs one verify read against the new table/column on prod, records the applied migration tag, and stops.

ClosesCloses the gap that caused the 2026-06-11 dashboard outage, where an unapplied migration meant every query on the altered table 404'd.
StopsTerminal states: No-op (nothing pending), Success (applied + verify query succeeds), Approval-required (DROP/RENAME/destructive ALTER/backfill SQL, or before any prod write — stop and ask), Blocked (migrate errors → surface, do not retry). Cannot run forever: at most one apply per trigger, no retry loop.
HelpsGuarantees that the altered table answers in prod immediately after a migration PR merges (verified by a self-contained read, not dependent on smoke), preventing dashboard-breaking unapplied-migration outages.

Post-merge smoke watch

Read-only

After a squash-merge lands on `main` (which auto-deploys prod), it captures the merged SHA and polls `gh run list --workflow=smoke.yml --branch main` for that SHA until the run's status is `completed`, then stops polling and reports the conclusion.

ClosesThe manual "verify the post-deploy smoke went green after every merge" rule — automating it so a red smoke becomes an immediate, specific report instead of a silent prod regression.
StopsThree terminal states: Success (conclusion `success` → report green, done); Blocked (conclusion `failure` → fetch `gh run view --log-failed`, surface the specific failing job, stop, and do not stack more merges on top); No-progress (deploy finished but no smoke run is ever created for the SHA → stop and report it didn't trigger). Cannot run forever — polling ends at the run's terminal state; one run watched per merge, never polling past completion.
HelpsTurns a red smoke into an immediate, specific report instead of a silent prod regression, making the post-deploy green-verification step automatic after every merge.

iCloud dup-file sweep

Local delete (PR)

In one bounded pass, lists the stray `* 2.*` iCloud-sync duplicate files (tracked via `git ls-files | grep -E ' 2\.(ts|tsx|sql|js|mjs)$'`, untracked via `find . -path ./node_modules -prune -o -name '* 2.*' -print`), removes each (`git rm` if tracked), `rm -rf .next`, verifies (code dups: `npm run build` stays green; non-code dups: confirm the real counterpart without ` 2` still exists and is tracked), records what was removed, then stops.

ClosesThe existing `find … -name '* 2.ts'` guard only matches `.ts` and misses `.tsx`, `.sql`, and `.mjs` — exactly the extensions that slip through and trigger `Duplicate identifier` build breaks, e.g. the `page 2.tsx` that forced cleanup PR #173 and the `0008_*_thrasher 2.sql` orphan.
StopsTerminal states: No-op (none found), Success (duplicates removed + build green), Blocked (build breaks after a removal ⇒ that file was the real one: restore it and stop). Cannot run forever — it is a single pass; if a removed duplicate reappears (iCloud re-sync), stop and surface it rather than sweeping again in a loop.
HelpsKeeps the build green by catching every duplicate extension before a gate/commit, preventing the `Duplicate identifier` breaks that iCloud-sync `* 2.*` files cause.

Release-gate integrity audit

Read-only

After a merge session or on demand, it reads main's branch protection (gh api repos/Nagavenkatasai7/claude-payments/branches/main/protection) and confirms strict required-checks is on, then checks that each merge since the last audit (fallback: last ~10) landed with its head SHA's `ci` check concluded success; it reports any drift (protection weakened, or a PR merged without a green `ci`) and stops after one bounded pass.

ClosesIt closes the silent gate-weakening that happens because landing PRs here repeatedly weakens protection — `gh api … required_status_checks -F strict=false` → merge → `strict=true` (the #171/#172 batch) — plus `gh pr merge --admin` on every merge; if the restore is skipped, protection stays weakened silently, an `--admin` merge can land a red/missing `ci`, and a stale-base merge can combine two individually-green PRs that break together.
StopsTerminal states: No-op (protection intact + every merge in the window green); Success (protection restored with approval and/or findings reported); Approval-required (before any protection write); Blocked (a merge landed on a red/missing `ci` → surfaced for a human to revert/forward-fix, never reported as success). Cannot run forever: a single pass over a bounded window plus one protection read.
HelpsIt keeps main's release gate intact so a forgotten `strict=true` restore or an `--admin` merge over a red/missing check can't silently hide, protecting the integrity of every merge into the deploy branch.

Dependency-CVE burndown

PR

On a schedule (or when npm audit flags a new high/critical), it runs `npm audit --json`, picks the highest-severity advisory that has a fix (one dependency at a time), applies the minimal bump, then verifies with `npx vitest run` + `npm run build` + re-audit — keeping the bump only if both pass and the advisory clears, else reverting; it stops when no high/critical advisories remain or all that remain are parked as "no non-breaking fix."

ClosesThe manual hand-bump of overnight CVEs: the nodemailer HIGH advisory forced a v6→v9 bump because ci.yml's `npm audit --omit=dev --audit-level=high` rejected it, and nightly.yml runs a full-tree audit to catch newly-published advisories — but the gate fails without fixing, so a CVE published overnight blocks the next unrelated PR until someone bumps by hand.
StopsTerminal states: No-op (no high/critical advisories), Success (green bumps opened as one PR), or Blocked/approval-required (an advisory with no fix that keeps tests green is marked for an approved-risk decision). Cannot run forever: one dependency per cycle, and an unfixable advisory is marked-and-skipped (never re-attempted in the same run), so every cycle makes progress (an advisory clears or is parked) and the run ends when none are actionable; revert-on-red keeps each step reversible.
HelpsBurns down high/critical npm advisories one safe, test-proven bump at a time before they turn an unrelated PR's CI red.
docs/loops/dependency-cve-burndown.md

Flaky-test stabilizer

PR

When a test fails in CI/nightly but passes in isolation, reproduces the nondeterminism by running that file repeatedly under the suite's real parallelism (bounded attempts), root-causes it (shared state, timer order, time-window fixture) and applies one fix without weakening assertions, then verifies with a fixed consecutive-pass streak (e.g. 10x, zero failures); one test per pass.

ClosesFlakes are currently masked rather than fixed: vitest has a CI-only retry, nightly.yml's comment names "flake accumulation," and CLAUDE.md warns "occasional parallel-run flakes pass in isolation" plus the freshDb()-before-vi.useFakeTimers() ordering trap — the retry hides nondeterminism instead of fixing it.
StopsTerminal states: No-op (no flaky test in scope); Success (streak holds with assertions intact → PR the fix); No-progress (can't reproduce after the attempt budget → report "couldn't reproduce", explicitly NOT success, do not loop). Cannot run forever: bounded reproduction attempts plus a finite consecutive-pass streak as the gate, one test per pass. Anti-gaming guard: a fix that skips the test or deletes assertions is forbidden — the streak must pass the full test or it's not success.
HelpsTurns masked flakes into genuinely fixed tests by reproducing and root-causing the nondeterminism, restoring trust in the CI/nightly suite instead of letting retries paper over accumulating flake.
docs/loops/flaky-test-stabilizer.md

Security-invariant audit

Read-only

A read-only sweep that lists the public surfaces changed since the last audit (server actions, `app/api/**/route.ts`, pages reading route params, via git diff), checks each against the repo's documented spine + security-pack invariant checklist, and emits per-surface findings (pass, or violation with file:line evidence); it stops after one pass over the changed-surface set.

ClosesThe manual pain of public surfaces silently failing to self-gate as features land: the partner-application flow needed token re-validation on 3 entry points (page, action, upload route); server actions are public POSTs that each must `require*` + validate scope; and the FIELD_ENCRYPTION_KEY boot-assert incident showed the boot-assert must mirror the accepting code's required secrets.
StopsBounded to the changed-surface set since the last audit, one pass, read-only with no fix loop, so it cannot run forever. Terminal states: No-op (every changed surface passes) or Success (findings reported, handed off to a human fix step).
HelpsCatches silent breaks of documented spine/security-pack invariants — self-gating (`require*` + scope, route params authoritative over body), server-side re-validation of token-gated routes (token → not-expired → not-used), money paths using the PII-scrubbing logger (`src/lib/log.ts`) not console, partner-scoped queries filtering by `partnerId`, the boot-assert mirroring every secret the accepting code requires, and per-IP rate limits + HMAC fail-closed on pay/rail/webhook surfaces — each finding citing reproducible file:line evidence.

Overnight Bug Hunt (invariant fuzzer for TDD'd pure helpers)

PR
🌙 1:00 AM ET

Fuzzes 8 curated pure helpers (fx, phone, field-crypto, dates, compliance, kyc-state-machine, id, ip-rate-limit) against their named invariants read-only; for each flaw that survives adversarial verification, writes a failing regression test plus the minimal fix and opens ONE PR — it stops with a clean no-op (no PR) if no flaw is found or none survives verification.

ClosesThe manual, unattended overnight work of edge-case fuzzing the project's TDD'd pure helpers against invariants the codebase already depends on — e.g. fx cross-rate must stay positive/finite with base-omission and round-trip tolerance (no NaN/Infinity on extreme/zero amounts), field-crypto must satisfy decrypt(encrypt(x))===x and REJECT tampered ciphertext (never silently mis-decrypt), compliance screening must be structurally untoggleable so no input shape bypasses it, the kyc-state-machine must never let a webhook alone auto-approve, and ip-rate-limit must fail-open on backend error.
StopsTerminates with no PR if zero flaws are found across the 8 targets (clean no-op), or if no flaw survives adversarial verification (none survived verify). Otherwise it opens exactly ONE PR and ends with the PR url; each Hunt agent finds at most ONE flaw per helper and verification defaults real=false when uncertain, so it can't loop unbounded — it makes a single bounded pass over the fixed target list.
HelpsCatches real invariant violations and crashes in the money-path pure helpers overnight, unattended, before they reach production — each fix lands as a red→green regression test plus minimal change behind the full gate (tsc/eslint/vitest/build) without weakening existing tests, all gated behind morning human review.

Claims-vs-Code Audit (overnight)

Read-only
🌙 2:30 AM ET

Extracts every concrete public product claim from the landing page (src/app/page.tsx), src/app/about/page.tsx, and docs/, then traces each to the enforcing code (file:line) and assigns a verdict — supported, narrow, mismatch, or unverifiable — adversarially re-checking only the "supported" verdicts to find a false case; it stops by writing a morning markdown report and opening a PR (or short-circuits with a "clean no-op" when no claims are found).

ClosesThe manual, never-done-on-schedule chore of checking whether SmartRemit's public marketing copy still matches what the code actually enforces — the high-risk claims it names: non-custodial / "never holds funds", sanctions screening always-on, the fee schedule, the corridor/country count, encryption-at-rest, "licensed partners settle", and delivery wording — where a drifted claim is a money/compliance/security liability for a money-movement company.
StopsTerminal states: (1) zero claims extracted → logs "clean no-op" and returns early; or (2) writes docs/loops/reports/claims-audit-<stamp>.md and opens the morning-report PR, returning the claim count, flagged count, and PR url. Bounded work: extraction is one pass over three fixed sources, the audit runs one verdict per extracted claim (a finite pipeline) with the re-check gated to only the "supported" subset, and it ends after a single PR is created — it never merges and never loops.
HelpsSurfaces copy-vs-code drift — especially HIGH-RISK mismatches first, each with claim text, code evidence, and a suggested fix (narrow the copy or fix the code) — as a reviewable morning PR, so a money company catches false public promises before a customer or partner holds them to one.

Prod-Health Triage (overnight)

Read prod (SELECT-only) + PR
🌙 4:00 AM ET

Overnight, it reads prod telemetry READ-ONLY (dead-letter outbox rows with their last_error, paid-but-undelivered transfers older than 15 min, OPEN compliance/KYC reviews older than 24h), root-causes and classifies each as code_fix / ops_flag / wont_fix (adversarially re-verifying every code_fix), then opens a single PR carrying the code fixes plus a report of ops items needing a human — and stops there, never merging.

ClosesReplaces the manual overnight sweep of prod health that the SmartRemit durability spine surfaces but leaves a human to chase: the outbox worker's dead-letter rows (attempts exhausted → deduped ops alert at 8), and reconcileSweep's stuck-paid (>15m) and stale-review (>24h) conditions that today only fire WhatsApp alerts to OPS_ALERT_PHONE. It turns those alerts into root-caused fixes/flags instead of a pager waking someone up.
StopsClean no-op and exits if the Observe step finds zero issues (or DATABASE_URL_READONLY is unset / the connection fails → returns issues:[] with a note); the Observe query is capped at ~25 issues total; otherwise it terminates at one pushed-but-unmerged PR on branch loop/prod-health-<stamp> and returns counts (issues / codeFixes / opsFlags / pr url). It never merges and never issues a production write.
HelpsEach morning the team gets a ready-to-review PR whose code fixes have already passed the full gate (delete ' 2.ts' dups, rm -rf .next, tsc --noEmit && eslint . && vitest run && npm run build) with a regression test, plus a docs/loops/reports/prod-health-<stamp>.md listing exactly the ops-flag items that need a human and the precise opsAction — shrinking overnight prod incidents to a triaged, verified queue without any risk of an unattended prod write.
